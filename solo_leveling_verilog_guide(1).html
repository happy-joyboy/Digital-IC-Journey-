Verilog for MIPS Architects

Since you're coming from MIPS and computer organization, I'd teach Verilog differently than I would teach someone who has never seen hardware.

You already know:

  • CPU
    • Registers
    • ALU
    • Control Unit
    • Memory
    • Clock

Verilog is how we describe those pieces.

Rule #1: A Verilog Module = A Hardware Block

Think of a module as a function, except it becomes hardware.

module adder(
    input [3:0] a, // 4-bit
    input [3:0] b, // 4-bit
    output [4:0] sum  // 5-bit
);

assign sum = a + b;

endmodule

Diagram:

+-------+ a --->| | | Adder |----> sum b --->| | +-------+

Example:
a = 0101 (5)
b = 0011 (3)
sum = 01000 (8)

Rule #2: Wires Carry Signals

Imagine physical copper wires.

wire x; or
wire [7:0] data; --> 8-bit wire:

data[7] data[6] ... data[0]

Rule #3: assign Means Continuous Connection

Example:

assign y = a & b;

Hardware:

a ----\ AND ---- y b ----/

Whenever A or B changes:
A=1 B=1 → Y=1
A=1 B=0 → Y=0
No CPU is running. The circuit simply exists.

Rule #4: Data Types — wire vs reg (and why it confuses everyone)

This trips up almost everyone coming from software.

  • wire -> a physical connection. Cannot store a value.
  • reg -> a variable that HOLDS a value between assignments.

reg does NOT mean "this becomes a register" in hardware. It just means "this is assigned inside an always block."

wire w;        // driven by assign
reg  r;        // driven inside always @(...)

Whether reg becomes an actual flip-flop or stays plain wires depends on how you write the always block, not on the keyword itself. This is Rule #7 below — it's the single most important rule in the whole language.

Other types you'll see:

  • integer -> 32-bit signed, used in loops/testbenches
  • real -> simulation only, not synthesizable
  • time -> simulation only

Rule #5: Blocking (=) vs Non-Blocking (<=)

Here's a rule you'll rely on constantly:

// Blocking — executes in ORDER, like normal code
always @(*)
begin
    y = a & b;
    z = y | c;      // sees the NEW value of y
end

// Non-blocking — all RHS evaluated first, THEN assigned together
always @(posedge clk)
begin
    q1 <= d;
    q2 <= q1;       // sees the OLD value of q1, not the new one
end
The rule everyone must memorize:
Combinational logic : (always @(*)) -> use blocking (=)
Sequential logic : (always @(posedge)) -> use non-blocking (<=)

Mixing them wrong is the #1 cause of simulation-vs-synthesis mismatches.

Rule #6: always @(*) for Combinational Logic

You've seen assign for combinational logic. There's a second way:

always @(*)
begin
    if(sel)
        out = b;
    else
        out = a;
end

@(*) means "re-evaluate whenever any input on the right-hand side changes." Same idea as assign, just usable with if/case instead of only ternary expressions.

⚠️ Danger — Inferred Latches: 
always @(*)
begin
    if(sel)
        out = a;
    // no else! -> out holds its OLD value when sel=0
    // this SYNTHESIZES A LATCH, probably not what you wanted
end
Rule of thumb: every combinational always block must assign every output on every path. Always have an else.

Rule #7: Blocking Style Decides What Hardware You Get

This is the deepest rule in the whole language.

  • always @(*) -> Combinational logic (gates, no memory)
  • always @(posedge clk) -> Sequential logic (flip-flops, memory)

Same-looking code, totally different hardware:

// Combinational: just a wire path
always @(*)
    y = a & b;

// Sequential: a flip-flop that SAMPLES a&b on the clock edge
always @(posedge clk)
    y <= a & b;

Verilog doesn't "build a CPU" from syntax — it builds hardware from this style choice, repeated thousands of times.

Build a Half Adder

A half adder adds two bits.

Example: 1 + 1 = 10
Output: Sum = 0, Carry = 1

Circuit:
Sum = A XOR B
Carry = A AND B

module half_adder(
    input a,
    input b,
    output sum,
    output carry
);

assign sum   = a ^ b;
assign carry = a & b;

endmodule

Build a Full Adder

Now include Carry-In: A + B + Cin

Example: 1 + 1 + 1 = 11
Outputs: Sum = 1, Carry = 1

module full_adder(
    input a,
    input b,
    input cin,
    output sum,
    output carry
);

assign sum = a ^ b ^ cin;
assign carry = (a & b) | (a & cin) | (b & cin);

endmodule

Combinational vs Sequential Logic

This is the biggest topic.

Combinational:

Output depends only on current inputs. No memory.
Example: assign y = a & b;

Sequential:

Output depends on past values. Memory exists.
Example:

always @(posedge clk)
begin
    q <= d;
end

This creates a flip-flop.

What Is a Flip-Flop?

Imagine a 1-bit memory cell.

D ---> [FF] ---> Q

At every clock edge: Q = D

Example:
Clock tick: D = 1 → Tick → Q becomes 1
Next tick: D = 0 → Tick → Q becomes 0

If Statements

Hardware can make decisions.

always @(posedge clk)
begin
    if(reset)
        count <= 0;
    else
        count <= count + 1;
end

Meaning: IF reset pressed clear counter ELSE increment

Your First Register

4-bit register:

module register(
    input clk,
    input [3:0] d,
    output reg [3:0] q
);

always @(posedge clk)
begin
    q <= d;
end

endmodule

Hardware: 4 Flip-Flops which together form a register.
Sound familiar? That's exactly what your MIPS registers are made of.

Counter Example

Let's build something useful.

module counter(
    input clk,
    output reg [3:0] count
);

always @(posedge clk)
begin
    count <= count + 1;
end

endmodule

Clock ticks: 0000 0001 0010 0011 0100 ...
This is one of the most common beginner circuits.

case Statements

if/else gets unreadable fast. This is how real ALUs and FSMs are written:

always @(*)
begin
    case(opcode)
        2'b00: result = a + b;
        2'b01: result = a - b;
        2'b10: result = a & b;
        2'b11: result = a | b;
        default: result = 0;   // ALWAYS include default
    endcase
end
opcode ---> [ CASE SELECT ] ---> result 00 01 10 11

This is exactly how you'll build your 4-bit ALU.

Multiplexer (Very Important)

A mux chooses between inputs.

  • sel=0 → choose A
  • sel=1 → choose B
assign out = sel ? b : a;

// Equivalent to:
if(sel)
    out = b;
else
    out = a;

Muxes appear everywhere in CPUs.

Finite State Machines (The Missing Middle Step)

FSMs are the bridge between counters and a CPU control unit, and they have a standard 3-part template:

module fsm(
    input clk,
    input reset,
    input go,
    output reg done
);

// 1. State declaration
parameter IDLE = 2'b00, RUN = 2'b01, DONE = 2'b10;
reg [1:0] state, next_state;

// 2. Sequential block — moves state forward on the clock
always @(posedge clk)
begin
    if(reset)
        state <= IDLE;
    else
        state <= next_state;
end

// 3. Combinational block — decides the NEXT state
always @(*)
begin
    case(state)
        IDLE: next_state = go ? RUN : IDLE;
        RUN:  next_state = DONE;
        DONE: next_state = IDLE;
        default: next_state = IDLE;
    endcase
end

always @(*)
    done = (state == DONE);

endmodule
go IDLE ────────> RUN ────────> DONE ^ │ └────────────────────────────┘

Every CPU control unit is an FSM built exactly this way.

Parameters (Reusable, Scalable Modules)

Your adder was hardcoded to 4 bits. Real designs never are:

module adder #(parameter WIDTH = 4) (
    input  [WIDTH-1:0] a,
    input  [WIDTH-1:0] b,
    output [WIDTH:0]   sum
);

assign sum = a + b;

endmodule

Instantiate it at any width:

adder #(.WIDTH(8)) my_adder (.a(x), .b(y), .sum(z));

This is how a "4-bit ALU" becomes a "32-bit ALU" without rewriting anything.

Module Instantiation (Building OUT OF Modules)

Everything above builds single blocks. Real design wires blocks together:

module top(
    input  [3:0] a, b,
    output [4:0] result
);

full_adder fa0 (
    .a(a[0]),
    .b(b[0]),
    .cin(1'b0),
    .sum(result[0]),
    .carry(c0)
);

// repeat for fa1, fa2, fa3 -> a 4-bit ripple-carry adder

endmodule
a[0] b[0] a[1] b[1] a[2] b[2] a[3] b[3] │ │ │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ ▼ ▼ [ FA0 ]───c0──►[ FA1 ]───c1──►[ FA2 ]───c2──►[ FA3 ] │ │ │ │ sum[0] sum[1] sum[2] sum[3]

This is literally a structural description — hardware built from smaller hardware, not logic equations. You've now seen all three styles: dataflow (assign), behavioral (always), and structural (module instantiation) — structural is how full CPUs get assembled.

generate Blocks (Skip Manually Instantiating 32 Things)

genvar i;
generate
    for(i = 0; i < 4; i = i + 1)
    begin : gen_fa
        full_adder fa (
            .a(a[i]), .b(b[i]), .cin(carry[i]),
            .sum(sum[i]), .carry(carry[i+1])
        );
    end
endgenerate

Same ripple-carry adder as above, but scales to any WIDTH automatically. Essential once you build a register file or ALU with repeated bit-slices.

Functions and Tasks

function [3:0] max_val;
    input [3:0] x, y;
    begin
        max_val = (x > y) ? x : y;
    end
endfunction
  • function -> combinational, returns exactly one value, no delays
  • task -> can have multiple outputs, can model timing (sim only)

Useful for repeated combinational logic you don't want to keep rewriting.

Concatenation and Replication (Bit Manipulation You'll Need Constantly)

wire [7:0] byte;
wire [3:0] hi = byte[7:4];
wire [3:0] lo = byte[3:0];

wire [7:0] rebuilt = {hi, lo};     // concatenation
wire [7:0] all_ones = {8{1'b1}};   // replication -> 11111111

You'll use this constantly for sign-extension, bus assembly, and instruction decoding (which your MIPS background will make click instantly — this is literally how you slice instr[31:26] for opcode).

Memories (Register Files, Instruction Memory)

module reg_file(
    input clk,
    input we,
    input [4:0] waddr, raddr1, raddr2,
    input [31:0] wdata,
    output [31:0] rdata1, rdata2
);

reg [31:0] regs [0:31];   // 32 registers, 32 bits each — a MIPS reg file

always @(posedge clk)
    if(we) regs[waddr] <= wdata;

assign rdata1 = regs[raddr1];
assign rdata2 = regs[raddr2];

endmodule

This is the actual Verilog behind "MIPS has 32 registers" — a 2D reg array.

Synchronous vs Asynchronous Reset

// Synchronous — reset only takes effect on clock edge
always @(posedge clk)
    if(reset) q <= 0; else q <= d;

// Asynchronous — reset takes effect IMMEDIATELY, ignores clock
always @(posedge clk or posedge reset)
    if(reset) q <= 0; else q <= d;

Real chips almost always use one or the other deliberately — worth knowing both exist and why.

What's Genuinely Simulation-Only (Won't Synthesize to Hardware)

Since testbenches are covered elsewhere, just know these exist only in simulation and will never become a real circuit — don't accidentally lean on them in a design module:

  • $display, $monitor, $time -> printing/debug
  • # delays (#5 a = 1;) -> fake timing, sim only
  • initial blocks -> testbench setup, not synthesizable hardware
  • real, time types -> sim only

Why This Matters for Digital IC Design

A typical learning path is:
Verilog Basics → Logic Gates → Adders → Multiplexers → Registers → Counters → Finite State Machines → ALU → Simple CPU → Pipeline CPU → ASIC/FPGA Design

Given your background, I'd actually skip some beginner fluff and move toward building:

  • Half Adder
  • Full Adder
  • 4-bit ALU
  • Register File (like MIPS)
  • Finite State Machine
  • Simple single-cycle CPU

That's where Verilog starts feeling less like a language and more like "I am constructing a processor from the ground up."
